## What landed
- **M0-2 complete.** `python3 -m pytest` from `ml-system/` → **17 passed** (4 config + 13 model).
  - `fitted_core/models.py` — the four deliverables:
    - `ItemType` enum (`top, bottom, dress, outer_layer, shoes`). Member **order is load-bearing**
      (R4 RNG-consumption order); member *names* = wire values so the M5 adapter needs no
      translation table. Lowercase members (matches session-doc F5's `ItemType.outer_layer`).
    - `WardrobeItem` dataclass — `id, name, type, warmth, image_url` required; tag lists default
      empty (`default_factory=list`); `material/formality` optional. `__post_init__` coerces
      `type` through `ItemType(...)` (rejects unknowns, accepts the M5 adapter's raw strings) and
      bounds `warmth` to 0–10.
    - `Template` (`one_piece`, `two_piece`) and `Role` (`base_top, base_bottom, one_piece,
      outer_layer, shoes`) enums — for M0-4's normalizer.
    - `SlotMap` dataclass — 5 `itemId | None` slots, all default `None`. Validity **not** enforced
      here (M0-4 owns it), so it can also hold the in-progress/rejected shapes the normalizer needs.
  - `tests/test_models.py` — plan's enumerated cases (with/without optionals, warmth 0 and 10,
    unknown type rejected, tags arbitrary strings) + R4 enum-order/membership guards + warmth
    out-of-range reject + wire-string coercion.
  - `tests/conftest.py` — `demo_wardrobe` fixture: legacy 8-item wardrobe (t1–t3, b1–b3, s1–s2)
    re-expressed in the new schema, FOOTWEAR→`shoes`. Legacy file untouched (decision §1.5-2).

## Decisions / why (typing-style, not design — escape hatch not tripped)
- **snake_case attributes** (`style_tags`, `image_url`) per PEP 8 / CLAUDE.md; spec's camelCase is
  wire-side, mapped in the M5 adapter. SlotMap's single-word slots are unaffected → no downstream
  collision.
- **Added `type` coercion + `warmth` range validation** beyond the bare field list. Makes the
  plan's boundary tests load-bearing rather than vacuous and catches M5-adapter bugs early; added
  matching reject tests.
- **Tags default empty** — an item legitimately has no tags ("no enum enforcement in v1"); only
  `material/formality` are the plan's "optional" fields.
